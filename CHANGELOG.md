# Changelog

All notable changes to `@julianoczkowski/agentwiki` are documented here.

## 1.1.0 ‚Äî 2026-07-16

### Everything anchors at the repo root

Running any `agentwiki` command from a monorepo subfolder used to create a second wiki (plus rules and a dead `.github` workflow) inside that subfolder. Now:

- **Git-root anchoring** ‚Äî every command resolves the git repository root (`git rev-parse --show-toplevel`) and creates/reads all artifacts there: `agentwiki/`, `.cursor/`, `.claude/`, `AGENTS.md`/`CLAUDE.md`, `.github/workflows/agentwiki.yml`. Outside a git repo, the current directory is used as before.
- **"You are here" pre-selection** ‚Äî running `init` from inside an app's folder pre-selects that app in the monorepo picker (labeled "you ran init from here"), so a single Enter scopes to it.
- **Visible anchoring** ‚Äî when invoked from a subfolder, the setup screen says `running from <dir>/ ‚Äî everything is created at the repo root`.
- **`doctor` detects orphaned nested wikis** ‚Äî wikis created in subfolders by pre-1.1 runs are reported with step-by-step cleanup hints.
- **README** ‚Äî new "How it works" Mermaid diagram showing the CLI ‚áÑ wiki ‚áÑ coding-agent loop and the GitHub Actions / editor-hook automation.

## 1.0.0 ‚Äî 2026-07-16

### Monorepo support üéâ

`init` now asks **"Which app should the wiki document?"** when it detects a monorepo, and the whole pipeline honors the choice.

- **Interactive app picker** ‚Äî a fresh `agentwiki init` in a monorepo opens an arrow-key select listing the detected applications (plus "The whole repository"). Single-project repos never see the question.
- **Apps vs shared packages** ‚Äî only *applications* are listed; shared packages/libraries (`packages/ui`, `libs/*`, tsconfig presets, ‚Ä¶) sit behind a "Something else‚Ä¶" option so the list stays clean. Classification uses NX `project.json` (`projectType`), `nx.json` `workspaceLayout`, and path heuristics, in that order.
- **Broad layout detection** ‚Äî npm/pnpm workspace globs, NX (including custom layouts like `clients/apps` and `project.json`-only apps), and a manifest sweep up to three levels deep that also covers Go, Rust, and Python monorepos with no JS workspace config.
- **Scoped facts everywhere** ‚Äî with a scope saved, scanning, symbol extraction, the module graph, and git facts (hot files, recent commits, 90-day counts) cover only that app. File paths in the wiki are app-relative; the quickstart carries a "Monorepo scope" fact and the enrich prompt tells your agent which app it is documenting.
- **`init --scope <dir>`** ‚Äî set or change the scope non-interactively (`--scope .` returns to whole-repo mode). The choice is stored in `agentwiki/.agentwiki.json`, so `update` ‚Äî including Cursor/Claude stop-hooks and CI ‚Äî respects it silently with no prompts.
- **Visible everywhere** ‚Äî `status` and `doctor` report the active scope.

All existing guarantees hold under a scope: repeated `update` stays a byte-identical no-op, commits to *other* apps never stale your prose, and prose written by your agent is never lost.

### Why 1.0.0

The `.agentwiki.json` metadata gained a `scope` field and `ScanResult`/`GitFacts`/`buildEnrichPrompt` signatures changed. Wikis generated by 0.x keep working unchanged ‚Äî whole-repo remains the default.
